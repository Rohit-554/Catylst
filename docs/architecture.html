Architecture & Setup

Full developer reference for building on top of Catylst.

Overview

Catylst is a fully-wired KMP project. Generate a fresh project from the IDE plugin or CLI — pick which of Android, iOS, and Desktop you want, and the rest is stripped out. Every built-in feature can also be removed later in minutes using the included Claude Code skills.

Built-in features: Compose Multiplatform UI, Koin DI, Room 3.1 database, Ktor networking, a Ktor server backend module, Navigation3, WorkManager / UNUserNotificationCenter notifications, cross-platform runtime permissions, multiplatform preferences, and provider-agnostic AI (Claude / Groq / Gemini).

Generate a Project

Two ways to scaffold a project. Both let you name your package and app, choose target platforms, select features, and pick an AI provider — no manual renaming required.

Option 1 — IDE Plugin (recommended)

Install Catylst KMP Starter from the JetBrains Marketplace (Android Studio / IntelliJ), then create a project through the wizard.

# In the IDE
Settings → Plugins → Marketplace → search "Catylst KMP Starter" → Install

# Then
File → New → Project… → Catylst KMP Starter

The wizard collects your package, app name, target platforms, features, and AI provider, then generates a ready-to-build project. Marketplace listing: plugins.jetbrains.com/plugin/31925-catylst-kmp-starter.

Option 2 — CLI

# Install
npm install -g catylst

# or via the install script
bash <(curl -sSL https://raw.githubusercontent.com/Rohit-554/Catylst/main/scripts/install.sh)

# Generate (interactive prompts for name, platforms, features, AI provider)
catylst --interactive

Option 3 — Clone & rename manually

If you'd rather start from the raw template, clone and run the setup script once. It renames everything — package, app name, source directories, iOS config — then deletes itself and the docs/ folder.

git clone https://github.com/Rohit-554/Catylst.git myapp
cd myapp
bash scripts/setup.sh com.yourname.yourapp YourApp
One-time only. After generating (any method), sync Gradle in your IDE, then build once to verify: ./gradlew :androidApp:assembleDebug.

Platform Selection

When generating from the plugin or CLI you choose any combination of Android, iOS, and Desktop. Unselected platforms are fully removed from the generated project — source sets, Gradle targets, dependencies, and modules — so you don't carry code you won't ship.

PlatformWhat ships
AndroidandroidApp/ module + androidMain source set
iOSiosApp/ Xcode project + iosMain source set
DesktopdesktopMain source set + Compose desktop entry point, runs with ./gradlew :composeApp:run
Desktop runs out of the box. Generated projects include a Compose desktop main.kt and compose.desktop application config — no extra setup needed.

Module Structure

YourApp/
├── androidApp/               # Android entry point only
│   └── src/main/kotlin/.../
│       └── MainActivity.kt   # Injects API keys, starts Koin
│
├── server/                   # Ktor server backend module (optional)
│
└── composeApp/               # ALL shared code
    └── src/
        ├── commonMain/kotlin/.../
        │   ├── ai/               # AiProvider + Claude/Groq/Gemini
        │   ├── config/           # AppConfig (API key holder)
        │   ├── data/
        │   │   ├── local/        # Room entities + DAOs
        │   │   ├── preferences/  # AppPreferences
        │   │   └── repository/
        │   ├── di/               # AppModule.kt - all Koin bindings
        │   ├── navigation/       # Screen.kt + AppNavigation.kt
        │   ├── network/          # HttpClientFactory + ApiService
        │   ├── notifications/    # expect NotificationScheduler
        │   ├── permissions/      # expect PermissionController
        │   └── ui/
        │       ├── screens/
        │       └── viewmodel/
        ├── androidMain/      # actual implementations for Android
        ├── iosMain/          # actual implementations for iOS
        └── desktopMain/      # actual implementations for Desktop
Rule: Put everything in commonMain. Only use platform source sets for actual implementations of expect declarations.

Navigation

Navigation3 with rememberNavBackStack() and Crossfade. Never use NavDisplay - it is not KMP-compatible.

Adding a screen

// 1. Screen.kt
@Serializable
data object Xxx : Screen()

// 2. AppNavigation.kt - SerializersModule
subclass(Screen.Xxx::class)

// 3. AppNavigation.kt - Crossfade branch
is Screen.Xxx -> XxxScreen(onBack = { backStack.removeLastOrNull() })

// 4. HomeScreen.kt - button
OutlinedButton(onClick = onNavigateToXxx) { Text("Open Xxx") }
Use the bloom-build Claude Code skill for the full guided walkthrough — it scaffolds the screen, navigation, ViewModel, and any Room/DI wiring in one pass.

Agent Skills

Selected skills install into .claude/skills/ of the generated project and are picked up automatically by Claude Code. Invoke them by name for guided, step-by-step changes tailored to this codebase.

bloom-build
Build screens end-to-end — composable UI, navigation wiring, ViewModel, Room Entity/DAO/Repository, and Koin DI, all scaffolded in one pass.
bloom-navigate
Modify the generated project — swap the active AI provider, add notification channels, configure runtime permissions, or cleanly remove any built-in feature.

Common Pitfalls

BuildConfig unresolved in composeApp
Lives in :androidApp only. Use AppConfig object in commonMain instead.
NavDisplay not found
Not KMP-compatible. Use Crossfade + rememberNavBackStack() as in AppNavigation.kt.
java.util.Properties in defaultConfig {}
Declare Properties at .kts file top-level, not inside defaultConfig block.
Room entity changes not picked up
Run ./gradlew :composeApp:kspAndroidMain after any Entity/DAO change.
Koin inject fails at runtime
Check binding exists in appModule() or platformModule().
buildConfigField value is empty
Use "\"${value}\"" format in .kts - bare quotes get stripped.
Inline qualified import
Add import at file top, use unqualified name in body.
KN 2.x ObjC method not found (iOS)
Use ClassName.Companion.method() instead of top-level call.
